Step 02 Branching Strategy

Create a Feature Branch

Always isolate new work — whether a bug fix or a new feature — in a dedicated branch. This keeps main stable and enables clean pull requests and peer review.

Create and switch to a new branch
terminal
$git checkout -b bugs/sqli-fix
Creates branch bugs/sqli-fix and switches to it immediately. The -b flag combines create + checkout in one command
Recommended branch naming conventions
  • bugs/ prefix — for defect and vulnerability fixes, e.g. bugs/sqli-fix
  • feature/ prefix — for new functionality, e.g. feature/auth-mfa
  • hotfix/ prefix — for urgent production patches, e.g. hotfix/null-pointer
  • refactor/ prefix — for code quality improvements without behaviour change
⚠ Never commit directly to main in a collaborative project. Every change — even a one-line fix — belongs on a dedicated branch to preserve a clean review and merge history.

Claude Code drafts the specification collaboratively. Always review every assumption and acceptance criterion before proceeding
Specification file lifecycle
  • Collaborate with Claude Code to produce the initial draft
  • Review all assumptions, edge cases, and acceptance criteria carefully
  • Finalise and save to vuln-app-cc/.claude/specs/sqli-fix.md
  • The spec file becomes the single source of truth for all subsequent planning and implementation
  • A good spec reduces AI hallucinations and misaligned implementations significantly

Step 07 Version Control

Commit and Push Changes

Once you have tested and validated the implementation locally, commit your changes with a descriptive message and push the feature branch to the remote GitHub repository. All shell commands can be run directly within Claude Code using the ! prefix.

Stage, commit, and push from within Claude Code
Claude Code · ! prefix runs shell commands
›! git add .
The ! prefix executes shell commands from within Claude Code. Stages all modified and new files
›! git commit -m "Fix SQLi vulnerability in login endpoint"
Write a clear, imperative-mood message: what changed and why. This message is permanent project documentation
›! git push origin bugs/sqli-fix
Push the feature branch to GitHub. This does not affect main — changes stay isolated on the feature branch
⚠ Write commit messages in imperative mood: "Fix…", "Add…", "Refactor…". Avoid vague messages like "update" or "fix bug". Good commit messages are the project's long-term documentation and invaluable during debugging.

Step 08 Pull Request

Open a Pull Request on GitHub

A Pull Request (PR) is a formal proposal to merge changes from your feature branch into the target branch. It initiates code review, discussion, and quality gates before changes reach main.

PR creation steps on GitHub
  • Navigate to the repository on GitHub.com — you will see a banner for the recently pushed branch
  • Click the green "Compare & pull request" button next to the branch banner
  • Set base branch to main — this is the merge target
  • Set compare branch to bugs/sqli-fix — this contains your changes
  • Write a clear title and a detailed description explaining what changed and why
  • Click "Create pull request" to formally submit your changes for team review

Step 09 Review & Merge

Review, Merge the PR, and Clean Up

Team members review the diff, discuss changes, and test locally. Once approved, the maintainer merges into main. Merge conflicts — when the same line has diverged in both branches — must be resolved before merging.

Post-merge local cleanup
Claude Code · post-merge
›! git checkout main
Switch back to the main branch locally
›! git pull origin main
Pull the newly merged changes — your local main now includes the fix
›! git branch -d bugs/sqli-fix
Delete the local feature branch — it has been merged and is no longer needed
Optional — Create and push a release tag
terminal · tagging a release
$git tag -a v1.0.0 -m "Release version v1.0.0"
Annotated tags bookmark a specific commit in Git history with a human-readable message
$git push origin v1.0.0
Push the tag to GitHub. Then navigate to Releases → "Create a new release" on the GitHub repo page
⚠ Keep branches short-lived. Long-lived branches accumulate divergence from main and produce difficult merge conflicts. Merge often, delete promptly. If main changes while you are working on a feature, run git checkout main && git pull origin main to sync before continuing.
